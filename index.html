springcloud学习

一、服务注册中心

euraka: AP,高可用分布式治理服务中心，服务注册，服务发现。
    集群：相互注册，相互守望
    dicovery获取服务的名称，IP，端口号等信息
    自我保护机制（不会马上删除故障服务）

zookeeper:CP，没有图形界面

consol:CP

nacos：springcloud-alibaba

二、负载均衡

ribbon: loadBalanced+restTemplate实现
    轮询算法，随机算法等，继承AbstractLoadBalanced类实现IRule接口
    轮询算法：请求次数%服务数，取余（discovery+乐观锁、自旋锁）

feign/openFeign:
    面向接口（通过配置FeignClient）
    feign已经停更
    本身已经集成ribbon负载均衡算法

三、熔断降级(hystrix)

服务降级：抛出异常，超时，宕机之后要有降级回调方法（PS：JMeter压力测试） 
    服务提供者：DefaultProperties，全局降级回调方法
    服务消费者：feign+yhstrix(@feignClient(...fallbace="")),创建一个hystrix类实现feignService接口

服务熔断：应对微服务雪崩效应的的一种链路保护机制，当某个服务频繁返回错误信息或超时，会进行服务降级，进而熔断该服务节点。
          服务降级->服务熔断—>尝试恢复
          配置：窗口时间内，达到请求次数的一定百分比数量失败后，断路器将半开，如果下一个请求还是失败，则熔断
          流程：构造hystrixCommand构造器->执行命令-> 查看缓存 -> 查看断路器是否打开(打开了则返回fallback) -> 
          查看线程池，信号量是否已满(满了则返回fallback) -> 执行方法hystrixCommand.run()（超时，异常）->
          hystrixCommand向断路器汇报状态并统计是否开启断路器 -> 降级

服务限流

四、网关
zuul 1.x:已停更，BIO模型（底层由servlet2.5实现），效率低

zuul 2.x:跳票，没推出（异步非阻塞+长连接）

GateWay:Springcloud整合推出，整合了spring5,webflux,netty等技术，异步非阻塞Reactor模式，效率高

    动态路由：构建网关的基本模块，由ID+URL组成，由一系列断言和过滤器组成
    断言（predicate）可以匹配HTTP请求中的内容（匹配条件）
    过滤(fliter) 对路由进行修改（拦截器）
    继承hystrix

五、配置中心

集中管理配置文件，不同环境不同配置（dev/test/beta...），运行期间动态更改配置文件，配置信息以REST接口暴露

application.yml是用户级配置文件，bootstrap.yml是系统级（优先级更高）

六、消息总线Bus

可以实现Springcloud config配置动态刷新，整合了java事件处理机制以及消息中间件，目前支持rabbitMQ和kafka

什么是总线？
    在微服务架构中，通常使用轻量级的消息代理来构建一个共用主题，并让系统中的所有微服务连接上，
    由该主题产生的消息会被所有实例监听以及消费

原理
    configServer实例监听MQ中同一个topic，当配置发生更新，会把消息放入topic中，其他服务会得到通知

七、消息驱动

屏蔽底层消息中间件的差异，降低切换成本，统一消息的编程模型，实现解耦

Springcloud Stream
    构建消息驱动的微服务框架。应用程序通过inputs(消费者),outputs(生产者)与springcloud stream中的binder对象进行交互
    通信方式遵循了发布-订阅模式

    binder: 连接中间件，屏蔽差异

    channel:通道，是队列的一种抽象，实现消息存储和转发的媒介

    source/sink:输入（接收消息）与输出（发布消息）

分组消费与持久化

    消费：存在重复消费的问题。不同组会重复消费，同一组内会有竞争关系（只有一个可以消费），配置group属性

    持久化：配置上group属性可以持久化，避免消息丢失

八、链路跟踪

在微服务框架中，每一个客户端请求都会经过多个不同服务节点调用来协同产生最后结果，形成一条负责的分布式调用链路

springcloud sleuth
    提供了一套完整的服务跟踪解决方案
    Trace:一条链路一个trace_id，唯一标识
    span:调用链路的来源，一次请求一个span
    parent:上一个请求的span

zipkin
    图形化展示

二、volatile

轻量级的同步机制

2.1 volatile特性

1.保证可见性(其中一个线程修改了volatile值，其他线程可以看到)
2.不保证原子性（number++） number++字节码：从内存中取值——》+1——》写入内存中，解决方案：1.sychronized 2.使用atomic原子类
3.禁止指令重排，多线程环境中，由于编译器优化重排的问题，无法保证变量的一致性（指令重排有数据依赖性）

内存屏障：是一个CPU指令，保证特定操作的执行顺序，保证某些变量的内存可见性（利用该特性实现volatile）

2.2 运用场景

DCL单例模式（双锁单例模式多线程指令重排情况下不安全），解决办法：加入volatile

原理：初始化对象分为3步，1.分配对象内存空间 2.初始化对象 3.设置对象指向分配的内存地址（instance!=null）

三、CAS

CompareAndSet 比较并交换，全称CompareAndSwap，是一条CPU并发原语（原语的执行必须连续，不会被打断 ），判断内存某个值是否为期望值，是则更新，这个操作是原子的

3.1 atomic原子类 compareAndSet(期望值，修改值)

底层原理：

Unsafe:CAS核心类，JAVA无法直接访问底层系统，需要通过native本地方法来访问，Unsafe相当于后门，可以直接操作特定内存的数据，Unsafe类根据内存地址偏移量来获取数据，Unsafe.getAndAddInt(对象，内存偏移量，更新值)，通过do…while实现自旋，提高了并发性

底层：compareAndSwapInt(Native方法)

缺点：
循环时间长，开销大（如果一直失败，会一直自旋，CPU开销大）
只能保证一个共享变量的原子操作
ABA：只管开头和结尾
解决方法：时间戳原子引用（版本号），用AtomicStampeReference

原子引用：AtomicReference(自定义原子类)

四、集合类线程不安全问题

4.1 ArrayList

多线程环境下报concurrentModificationException，多线程情况下操作list出现抢占的情况，一个线程正在写，其他线程抢占

解决方案：
1.用vector
2.用collections类的sychronizedList包装
3.CopyOnWriteArrayList 写时复制，读写分离思想

4.2 Set

底层是hashmap，add方法为map.put(key,object常量)
不安全，多线程环境下报concurrentModificationException

解决方案：
1.用collections类的sychronizedSet包装
2.CopyOnWriteSet->底层是CopyOnWriteArrayList

4.3 Map

不安全，多线程环境下报concurrentModificationException

解决方法：
1.用collections类的sychronizedMap
2.conCurrentHashMap
